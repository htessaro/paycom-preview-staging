title: Refunds
description: Return settled funds back to the customer's original payment method

The Refunds API allows you to return funds from a previously created [charge](/docs/payments/04-payment-concepts/03-charges) back to the original payment method. 
This is a programmatic, backend-only flow used to manage reversals for settled payments.

The system supports highly flexible refund logic to accommodate various business needs:

* **Partial and multiple refunds:** Pay.com supports partial refunds, allowing you to return only a portion of the original charge amount.
* **Sequential reversals:** You can execute partial refunds multiple times until the entire original charge amount has been completely refunded.
* **Balance tracking:** When you retrieve a charge, the system provides an `amount_refunded` parameter so you can calculate the remaining balance available to be refunded.
* **API requirements:** To process a refund via the API, you must send a request containing the specific `charge_id` and the `amount` to be refunded.

## Retrieving refunds

The API provides several methods for querying and tracking refund data:

* **Retrieve a specific refund:** You can query the API with the unique identifier of a refund (returned from your initial creation request) to retrieve the details of that specific refund object.
* **List all refunds:** You can use the bulk query endpoint `GET /v1/refunds` to retrieve a list of your refunds. This endpoint returns an array of refund objects and supports query parameters for search and pagination, such as `limit` (up to 100 records), `starting_after`, and `ending_before`.
* **Charge timeline and sub-objects:** You can view an array of all refunds linked to a specific charge by retrieving that charge's details. This allows you to track each individual refund interaction associated with the original transaction.

## Webhook events

Because refund processing communicates with external banking networks, you should rely on webhook events to track the final outcome of your refund requests.

A refund resource triggers the following specific webhook events:

* `refund.succeeded`: Sent when a refund is successfully created for a charge.
* `refund.failed`: Sent when a refund request has failed.
* `charge.refunded`: Sent when a charge is successfully refunded, whether fully or partially. The event payload includes a boolean parameter called `refunded` which indicates whether the original charge has been fully refunded or not. Additionally, specific refund details can be accessed through the `refunds` sub-object within this event.
