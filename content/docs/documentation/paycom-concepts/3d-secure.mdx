3D Secure is a standard authentication protocol used to verify a customer's identity during a transaction. Pay.com provides an **Agnostic 3DS** layer: an authentication layer that runs at the gateway level, independently of the specific bank or processor. This lets you authenticate a customer once and route the authorized transaction to any processor for execution.

The system is designed to handle both frictionless flows and active challenges, where the user must verify the payment through their banking app or an SMS code.

## SDK integration

If you're using the Pay.com SDKs, Pay.com handles the entire authentication flow for you. The SDK automatically detects if a card needs extra verification and manages the entire process in the background:

* The SDK automatically determines if a card requires 3DS based on your specific configuration and the bank's requirements.
* It handles the rendering of the challenge window and captures all user interaction events directly.
* It completes the authentication process and moves to the payment result without requiring you to manage manual redirections.

## API integration

If you prefer using the direct API, you have full control over the user experience, but you
need to manually guide the customer through the bank’s verification steps.

<Steps>
<Step>
### Start the transaction

Send a `POST` request to [`/v1/charges`](/api-reference/charges/create-charge) or
[`/v1/holds`](/api-reference/holds/create-hold). You must include the
`authentication_context` object, which includes browser details, user agents, and IP addresses.
The endpoint references above list all accepted fields for `authentication_context`.

<Callout type="info">
Passing the customer’s email in the `billing_details` object maximizes 3DS approval rates.
</Callout>
</Step>
<Step>
### Check the status

If the bank requires a challenge, the API returns a transaction status of
`requires_authentication`. If not, the transaction finalizes immediately as `succeeded` or
`failed`.
</Step>
<Step>
### Create an authentication session

Send a `POST` request to
[`/v1/sessions/authentication/linked`](/api-reference/authenticationsessions/create-linked)
using the ID of the transaction from the previous step. This generates a `url`. You can
present the challenge to the customer in one of two ways:

- **Iframe:** Render the `url` in an iframe (at least 600 × 400 px) on your page. The
  issuer’s verification UI appears without the customer leaving your site.
- **Full-page redirect:** Redirect the customer to the `url` directly.

When the cardholder completes the challenge, they are redirected according to the `confirm`
setting from the next step.
</Step>
<Step>
### Confirm the payment

The `confirm` parameter controls how Pay.com handles the final authorization:

- `confirm: true`: Pay.com automatically authorizes the payment once the challenge completes
  and notifies you via webhooks.
- `confirm: false`: The transaction moves to `requires_confirmation`. Wait for the customer to
  redirect back to your site, then trigger the final authorization via
  [`POST /v1/charges/{charge_id}/confirm`](/api-reference/charges/confirm-charge) or
  [`POST /v1/holds/{hold_id}/confirm`](/api-reference/holds/confirm-hold).
</Step>
</Steps>

## Standalone authentication

If you need to authenticate a cardholder before creating a charge, for
example during subscription enrollment or deferred payment flows, Pay.com
supports an independent authentication session that runs entirely outside of
the charge lifecycle. You receive the ECI and CAVV from the session and pass
them to the charge later.

<Callout type="info">
See [Standalone 3DS authentication](/docs/risk-disputes-fraud/standalone-3ds-authentication)
for a step-by-step integration guide.
</Callout>

## External 3DS

If you already perform 3DS authentication through third-party software, Pay.com supports an **External MPI** (Merchant Plug-In) flow. This lets you maintain your existing security setup while still using Pay.com's orchestration engine.

To use this flow, pass your pre-authenticated data (such as `ds_transaction_id`, `cavv`, or `eci`) directly to the `POST /v1/charges` API via the `3DS` object. Pay.com passes these details to the issuer to authorize the transaction without triggering a second challenge for the customer.

## Benefits of agnostic authentication

Using a gateway-level authentication layer offers several advantages for managing global payments effectively, such as:

* You can retry a failed authorization with a different processor using the same 3DS credentials because authentication is not tied to a specific acquirer.
* The system supports the latest 3DS2 standards to favor frictionless authentication whenever the bank allows it.
* The SDK hides the complexity of device fingerprinting and challenge rendering, saving your team significant time and effort.
