A refund returns funds from a completed charge back to the customer's original payment method.
Pay.com supports full refunds, partial refunds, and multiple refunds on the same charge.

## Creating a refund

To create a refund, call `POST /v1/refunds` with the charge ID and optionally an amount. You
can also issue refunds directly from the Pay.com dashboard by navigating to the transaction and
clicking **Refund**.

| Parameter | Required | Description |
|---|---|---|
| `charge` | Yes | The ID of the charge to refund. |
| `amount` | No | The amount to refund in the smallest currency unit. Omit for a full refund. |
| `reference` | No | Your custom reference string for this refund. |
| `metadata` | No | Custom key-value pairs to attach to the refund. |

For a step-by-step walkthrough, see the
[Refund a charge](/docs/payments/server-to-server-guides/refund-a-charge) guide.

## Full, partial, and multiple refunds

Pay.com gives you three ways to return funds from a single charge. A **full refund** returns
the entire charge amount, simply omit the `amount` parameter. A **partial refund** returns
less than the full charge, specify a smaller `amount`. You can also issue **multiple refunds**
against the same charge, as long as the total refunded amount doesn't exceed the original
charge amount. Each refund creates a separate refund object.

## Tracking refunded amounts

The charge object provides two fields to help you track how much has been refunded. The
`amount_refunded` field shows the total amount refunded so far (the sum of all refunds), and
the `refunds.data` array contains all individual refund objects linked to the charge. To check
how much is still available to refund, subtract `amount_refunded` from `amount`.

<Callout type="info">
The `refunded` field on the charge object is `true` only when the charge has been **fully**
refunded, meaning `amount_refunded` equals `amount`.
</Callout>

## Refund statuses

A refund has two possible terminal states:

| Status | Meaning |
|---|---|
| `succeeded` | The refund was processed successfully. Funds are being returned to the customer. |
| `failed` | The refund could not be processed. |

## Key fields on the refund object

When you retrieve a refund via `GET /v1/refunds/{refund_id}`, the response includes these
important fields:

| Field | Description |
|---|---|
| `amount` | The refund amount in the smallest currency unit. |
| `charge` | The ID of the charge this refund was issued against. |
| `status` | The current status of the refund (`succeeded` or `failed`). |
| `reference` | Your custom reference, if provided when creating the refund. |
| `metadata` | Custom key-value pairs attached to the refund. |

## Webhook events

Pay.com sends webhook events as a refund progresses through its lifecycle:

| Event | Trigger |
|---|---|
| `refund.succeeded` | The refund was processed successfully. |
| `refund.failed` | The refund failed. |
| `charge.refunded` | Sent on the charge when a refund is processed against it. |

## Refunds vs. hold cancellations

Refunds and hold cancellations serve different purposes. A **refund** applies after a charge
has been captured, funds are returned to the customer, and both partial and multiple refunds
are supported. A **hold cancellation** applies before a hold has been captured, the
authorization is released, but only full cancellation is supported. If you need to release only
part of the reserved funds on a hold, use a partial capture instead; the uncaptured amount is
released automatically. See
[Holds and captures](/docs/paycom-concepts/holds-and-captures) for more.
