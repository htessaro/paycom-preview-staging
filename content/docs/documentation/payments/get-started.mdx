Pay.com uses an orchestration workspace to manage your transactions, routing, and logic. Your choice of integration path determines how much you personalize the UI and defines your level of PCI responsibility.

<Steps>
<Step>
### Choose your integration path

Select the connection path that fits your business model and PCI-DSS compliance requirements:

| Integration type | Best for... | PCI Level |
| :--- | :--- | :--- |
| **Universal SDK** | Pre-built UI for card collection and 3DS. | **SAQ-A**  |
| **Headless SDK** | Full UI control. Pay.com handles the logic. | **SAQ-A-EP**  |
| **Tokenization** | Collect tokens via SDK and process on backend. | **SAQ-A-EP** |
| **API-only** | Direct server-to-server processing. | **Level 1**  |

<Callout type="info">
See our [Integration types](/docs/introduction/choose-your-integration) page for detailed comparison.
</Callout>
</Step>

<Step>
### Configure your workspace

Set up your organizational structure in the [Pay.com Dashboard](https://dashboard.pay.com) before you process payments.

* **Workspaces:** Organize your business by brand, country, or specific operational needs.
* **Settlement accounts:** Configure accounts for each provider or acquirer. 
* **Orchestration:** Share multiple settlement accounts within a single workspace to route payments across different providers.

<Callout type="info">
Pay.com takes care of the heavy lifting. Our team will configure your settlement accounts and routing connections for you during the onboarding process before you go live.
</Callout>
</Step>

<Step>
### Obtain API credentials

Regardless of your integration path, you will need an API key to authenticate your backend requests.

1. Go to **Developers** → **API keys** in the [Dashboard](https://dashboard.pay.com).
2. Copy your **API Key**. Test keys begin with `test_` and live keys begin with `live_`.
3. Pass your key via the `x-paycom-api-key` HTTP header in every request.
4. Select your [environment](/docs/api-reference/environments) Base URL:
   * **Production:** `https://api.pay.com`
   * **Staging:** `https://api.staging.pay.com`

With your credentials secured, you’re ready to start your integration.

<Callout type="warn">
Keep your API Key private. Never share it or commit it to repositories.
</Callout>
</Step>

<Step>
### Make your first request

Your first API call depends on the integration path you chose in Step 1:

#### For SDK (Payment sessions)
If you use the Universal SDK or Headless SDK, start by creating a payment session via a `POST` request to the [`/v1/sessions/payment`](/docs/api-reference/paymentsessions/create-ps) endpoint.
Include the `amount`, `currency`, and `customer_id` in your payload to receive the session parameters required to initialize the SDK on your frontend.

<Callout type="warn">
We recommend also passing the customer's `email` in this request. While not required to open a session, it is needed for 3D Secure authentication and prevents the SDK from prompting the user for it during checkout.
</Callout>

Follow the [set up the Pay.com SDK](/docs/payments/sdk/integrate-the-pay-com-sdk) guide to install the package and initialize your first checkout.

#### For Tokenization (Hybrid flow)
If you are using the SDK to tokenize cards, send a `POST` request to the [`/v1/sessions/tokenization`](/docs/api-reference/tokenizationsessions/create-tokenization-session) endpoint. Use the returned token to render the SDK, save the card, and charge on your backend.

See the [tokenize a card](/docs/payments/sdk/tokenize-a-card) guide for the full flow.

#### For API-only (Server-to-server)
If you maintain PCI-DSS Level 1 compliance, skip the session creation and process the transaction directly. Send a `POST` request to either the [`/v1/charges`](/docs/api-reference/charges/create-charge) endpoint (for immediate capture) or the [`/v1/holds`](/docs/api-reference/holds/create-a-hold) endpoint (to authorize now and capture later).
Include either raw card details or an external token in your payload to start the transaction.

See the [create a charge](/docs/payments/guides/create-a-charge) guide for the full API flow.

<Callout type="info">
Pass a unique V4 UUID in the [`idempotency-key` header](/docs/api-reference/idempotency-key) for all `POST` requests. This lets you safely retry requests after a network disruption without risking a duplicate charge.
</Callout>
</Step>
</Steps>

## Next steps

Now that you have configured your workspace and authenticated your first request, explore these resources to build out your integration:

* Read the [integration types](/docs/introduction/choose-your-integration) page for a detailed comparison of SDK-based and API-only flows.
* Follow the [set up the Pay.com SDK](/docs/payments/sdk/integrate-the-pay-com-sdk) guide to install the SDK and run your first checkout.
* Read the [create a charge](/docs/payments/guides/create-a-charge) guide to process your first payment via the API or SDK.
* Visit the [webhooks](/documentation/payments/webhooks) section to receive real-time notifications for payment successes, failures, and chargebacks.
* Read the [idempotency key](/docs/api-reference/idempotency-key) reference to make sure your integration handles retries safely.
* Explore the [API reference](/docs/api-reference/index) for a complete list of endpoints, including customers, subscriptions, and specialized payment methods.
