Pay.com uses an orchestration workspace to manage your transactions, routing, and logic. Your choice of integration path determines how much you personalize the UI and defines your level of PCI responsibility.

<Steps>
<Step>
### Choose your integration path

Select the connection path that fits your business model and PCI-DSS compliance requirements:

| Integration type | Best for... | PCI Level |
| :--- | :--- | :--- |
| **Universal SDK** | Pre-built UI for card collection and 3DS. | **SAQ-A**  |
| **Headless SDK** | Full UI control. Pay.com handles the logic. | **SAQ-A-EP**  |
| **Tokenization** | Collect tokens via SDK and process on backend. | **SAQ-A-EP** |
| **API-only** | Direct server-to-server processing. | **Level 1**  |

<Callout type="info">
See our [Integration types](/docs/introduction/choose-your-integration) page for detailed comparison.
</Callout>
</Step>

<Step>
### Configure your workspace

Set up your organizational structure in the [Pay.com Dashboard](https://dashboard.pay.com) before you process payments.

* **Workspaces:** Organize your business by brand, country, or specific operational needs.
* **Settlement accounts:** Configure accounts for each provider or acquirer. 
* **Orchestration:** Share multiple settlement accounts within a single workspace to route payments across different providers.

<Callout type="info">
Pay.com takes care of the heavy lifting. Our team will configure your settlement accounts and routing connections for you during the onboarding process before you go live.
</Callout>
</Step>

<Step>
### Obtain API credentials

Regardless of your integration path, you will need an API key to authenticate your backend requests.

1. Go to **Developers** → **API keys** in the [Dashboard](https://dashboard.pay.com).
2. Copy your **API Key**. Test keys begin with `test_` and live keys begin with `live_`.
3. Pass your key via the `x-paycom-api-key` HTTP header in every request.
4. Select your [environment](/docs/api-reference/environments) Base URL:
   * **Production:** `https://api.pay.com`
   * **Staging:** `https://api.staging.pay.com`

With your credentials secured, you’re ready to start your integration.

<Callout type="warn">
Keep your API Key private. Never share it or commit it to repositories.
</Callout>
</Step>

<Step>
### Make your first request

Your first API call depends on the integration path you chose in Step 1:

#### For SDK (Payment sessions)
If you use the Universal SDK or Headless SDK, start by creating a payment session via a `POST` request to the [`/v1/sessions/payment`](/docs/api-reference/paymentsessions/create-ps) endpoint. 
Include the `amount`, `currency`, and `customer_id` in your payload to receive the session parameters required to initialize the SDK components in your frontend application. 

<Callout type="warn">
We highly recommend also passing the customer's `email` in this request. While not strictly required to open a session, it is vital for 3D Secure authentication and prevents the SDK from needing to manually prompt the user for it.
</Callout>

#### For Tokenization (Hybrid flow)
If you are using the SDK to tokenize cards, send a `POST` request to the [`/v1/sessions/tokenization`](/docs/api-reference/tokenizationsessions/create-tokenization-session) endpoint. This is a simple API that requires no parameters in the payload [cite: 271, 272]. Use the returned token to render the SDK, save the card, and transact on your backend.

#### For API-only (Server-to-server)
If you maintain PCI-DSS Level 1 compliance, skip the session creation and process the transaction directly by sending a `POST` request to either the [`/v1/charges`](/docs/payments/server-to-server-guides/create-a-charge) endpoint (for immediate capture) or the [`/v1/holds`](/docs/api-reference/holds/create-a-hold) endpoint (to authorize now and capture later) [cite: 282, 284, 291]. 
Include either raw card details or an external token in your payload to initiate the transaction immediately.

<Callout type="info">
We recommend you pass a unique V4 UUID in the [`idempotency-key` header](/docs/api-reference/idempotency-key) for all `POST` requests. This ensures that if a network disruption occurs in transit, you can safely retry the request without risking double-charging the customer [cite: 374, 375].
</Callout>
</Step>
</Steps>

## Next steps

Now that you have configured your workspace and authenticated your first request, explore these resources to build out your integration:

* Use our **[Integration](/docs/introduction/choose-your-integration)** page to deep-dive into the technical differences between SDK-based and API-only flows.
* Visit the **[Webhooks](/documentation/payments/webhooks)** page to learn how to receive real-time notifications for payment success, failures, and chargebacks.
* Read about **[Idempotency](/docs/api-reference/idempotency-key)** to ensure your integration safely handles retries without creating duplicate transactions.
* Explore our **[API Reference](/docs/api-reference/index)** for a complete list of endpoints, including customers, subscriptions, and specialized payment methods.
