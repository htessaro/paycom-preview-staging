The Pay.com SDK is a core library that you can integrate directly into your web applications to provide secure, PCI-compliant payment and checkout experiences.

The SDK supports three integration modes. Each mode uses the same underlying package but is initialized and used differently depending on your use case, UI requirements, and PCI compliance level. 

## Universal SDK

The Universal Component is a fully hosted, customizable UI component. Pay.com renders the payment form inside your page, including field validation, error handling, and redirect-based payment methods (opened in a popup to preserve page context).

Use this mode when you want the fastest path to a working checkout and don't need to control the look and feel of the payment form.

## Headless SDK

The Headless SDK gives you full control over your checkout UI. You build and own the form; the SDK captures sensitive card data and handles all payment method logic behind the scenes.

Use this mode when your design requirements go beyond what the Universal Component supports, or when you need to iterate on UI independently of the payment integration.


## Tokenization 
Tokenization is a hybrid approach that combines the SDK on the client side with API calls on the server side. It lets you securely store a customer's card details without processing a payment at that moment.

### Tokenization limitations

Tokenization focuses on securely storing card details for later use. Because no payment is processed at this stage, the following limitations apply:
<Callout type="warn">
  Review these limitations before choosing the Tokenization mode.
</Callout>

- **Cards only.** Tokenization Sessions do not support Apple Pay, Google Pay, or any other payment method. Only card tokenization is currently supported.
- **No direct payment.** Tokenization does not process a payment. You must make a separate server-side API call to transact with the stored token.
- **Manual 3DS handling.** If the card requires authentication, you must handle the 3DS challenge yourself. This is not handled automatically.


## Choosing an integration mode

The table below compares the available integration modes. Your choice determines how much control you have over the checkout interface and payment flow.

| | Universal | Headless | Tokenization |
|---|---|---|---|
| **Description** | Hosted checkout with a ready-made UI | Fully customizable checkout using your own UI | Store card details for future use without processing a payment |
| **UI ownership** | Pay.com renders and manages the UI | You build and control the UI | You build and control the UI |
| **PCI scope** | Lowest | Medium | Medium |
| **Payment processing** | Yes | Yes | No |
| **Supported payment methods** | All supported payment methods | All supported payment methods | Cards only |
| **3DS / authentication** | Handled automatically by the SDK | Handled automatically by the SDK | Must be handled manually |
| **Session type** | Payment Session | Payment Session | Tokenization Session |
