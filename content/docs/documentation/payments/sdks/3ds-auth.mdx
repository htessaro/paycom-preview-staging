---
title: Verify a payment with 3DS
description: Complete a payment that requires customer authentication by creating a linked authentication session and confirming the transaction after the customer verifies with their bank.
---

import { Tab, Tabs } from 'fumadocs-ui/components/tabs'
import { Step, Steps } from 'fumadocs-ui/components/steps'

When a charge, hold, or payment method returns `requires_authentication`, the 
customer's bank is requesting 3DS verification before the transaction can be 
completed. 

3DS (3D Secure) is an additional security layer for card payments. When triggered, 
the customer is presented with a challenge from their bank to confirm they are the legitimate cardholder 
before the payment can go through.

This guide walks through how to create an authentication session, present the 
challenge to the customer, and confirm the transaction once authentication is complete.

<Steps>
  <Step>
    ## Create a linked authentication session

    Create a linked authentication session using the `id` of the charge, hold, or payment method that returned `requires_authentication`. Pass this identifier as the `resource` value in the request body, then send a `POST` request to `/v1/sessions/authentication/linked`:

   ```json
    {
        "resource": "pm_card_291650715036681216",
        "return_url": "https://yoursite.com/checkout",
        "confirm": false
    }
    ```

    | Field | Description |
    |---|---|
    | `resource` | The ID of the charge, hold, or payment method that requires authentication|
    | `return_url` | The URL the customer is redirected to after completing the bank verification |
    | `confirm` | Controls whether Pay.com automatically confirms the transaction after authentication |

    Set `confirm: true` to have Pay.com confirm the transaction automatically after authentication and then redirect the customer to your `return_url`. Set `confirm: false` to receive the redirect first and confirm the transaction from your server by calling the `/confirm` endpoint. The server-side approach is safer and required for payment methods.

    A successful response contains the following values:
    - `url`: The challenge page where the customer completes authentication.
    - `client_secret`: Use this to render the authentication with the SDK.
  </Step>

  <Step>
    ## Present the challenge to the customer

    You have three options for how to show the bank challenge to the customer:

    <Tabs items={['Redirect', 'SDK modal (Beta)', 'iframe']}>
      <Tab value="Redirect">
        Redirect the customer directly to the `url` returned from the linked authentication session. The customer completes the bank challenge on that page.

        When authentication is complete, the customer is redirected back to the `return_url` you provided. 
    </Tab>

      <Tab value="SDK modal (Beta)">
        Use the SDK to render the bank challenge in a modal overlay. The customer completes the challenge without leaving your page.

        Pass the `client_secret` from the linked authentication session to `pay.authentication()`:

        ```javascript
        await pay.authentication({
          clientSecret: '{client_secret}',
        })
        ```
       The SDK displays the bank challenge in a modal and resolves once the verification is complete, regardless of whether the customer passed or failed the verification. Once it resolves, proceed to confirm the transaction.
      </Tab>

      <Tab value="iframe">
        You can embed the bank challenge `url` in an iframe on your checkout page. The customer completes the challenge inside the iframe without leaving your page.
        When authentication is complete, the iframe returns to the `return_url` you provided. Use this to detect completion, close the iframe, and proceed to confirm the transaction. 
      </Tab>
    </Tabs>
  </Step>

  <Step>
    ## Confirm the transaction

    After the customer completes the bank challenge, confirm the transaction using the endpoint that matches the resource you initially created.
    <Tabs items={['Charge', 'Hold', 'Payment method']}>
      <Tab value="Charge">
        ```
        POST /v1/charges/{charge_id}/confirm
        ```
    The confirm request returns one of two outcomes:
        | Outcome | Description |
        |---|---|
        | `succeeded` | The charge completed successfully |
        | Payment method error | Authentication failed or the card was declined |
      </Tab>

      <Tab value="Hold">
    ```
    POST /v1/holds/{hold_id}/confirm
    ```
    The confirm request returns one of two outcomes:
        | Outcome | Description |
        |---|---|
        | `succeeded` | The hold completed successfully |
        | Payment method error | Authentication failed or the card was declined |
      </Tab>

      <Tab value="Payment method">
    ```
    POST /v1/payment_methods/{payment_method_id}/confirm
    ```
    The confirm request returns one of two outcomes:
        | Outcome | Description |
        |---|---|
        | `created` | The payment method was saved successfully |
        | Payment method error | Authentication failed or the card was declined |
      </Tab>
    </Tabs>
  </Step>
</Steps>
