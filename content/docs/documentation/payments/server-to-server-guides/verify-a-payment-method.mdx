Verifying a payment method lets you validate and vault a customer's card without processing an immediate charge. The system performs a zero-amount transaction to confirm the card with the issuer and stores it for future use.

Use this flow when a customer adds a card to their profile, or when you need to vault a card before the first recurring charge in a subscription flow.

<Callout type="warn">
Raw card data passes through your servers. This flow requires PCI Level 1 or SAQ-D compliance.
</Callout>

## Requirements

* Access to your [API credentials](/docs/api-reference/authentication).
* A `customer_reference_id` to associate the card with a customer record.
* Browser metadata (IP address, user agent, screen dimensions) if the issuer requires a 3DS challenge.

<Steps>
<Step>
## Create the payment method

Call the [`POST /v1/payment_methods`](/docs/api-reference/paymentmethods/create-payment-method) endpoint with the card details and `setup_future_usage: "off_session"` to vault the card for future off-session charges.

The API returns one of three outcomes:

* `created` — the card was verified and vaulted immediately. No further steps needed.
* `requires_authentication` — the issuer requires a 3DS challenge. Continue to the next step.
* error — card verification failed.

## Create an authentication session

This step is only needed if step 1 returned `status: "requires_authentication"`.

Call the [`POST /v1/sessions/authentication/linked`](/docs/api-reference/authenticationsessions/create-linked-authentication-session) endpoint with the payment method ID in the `resource` field. You must set `confirm` to `false` — there is no auto-confirm option for the payment method flow.

Calling this endpoint before the customer completes the bank challenge returns `{ "code": "card_declined", "decline_code": "authentication_failure" }`. This is expected — wait for the customer to be redirected to your `return_url` first.
</Callout>

<Step>
## Listen for webhooks

Subscribe to these events to track the outcome of each verification:

* `payment_method.created`: The card was verified and vaulted successfully.
* `payment_method.failed_authentication`: The verification was declined.

<Callout type="warn">
Always verify webhook signatures using the `Pay-Signature` header and your endpoint's secret key (via HMAC SHA-256) to ensure the event is legitimate.
</Callout>

</Step>
