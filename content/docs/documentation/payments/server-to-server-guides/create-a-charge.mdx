The [create a charge](/docs/api-reference/charges/create-charge) method lets you move funds from a customer to a merchant account. This API-only flow provides control over the checkout experience.
It is ideal for businesses that are fully PCI-DSS compliant and process payments using stored tokens or saved payment methods.

## Requirements

Before following the steps described in this guide, you must:

* [Select your environment Base URL](/docs/api-reference/environments).
* [Have access to your API credentials](/docs/api-reference/authentication).
* [Generate a unique `idempotency-key` header for each request](/docs/api-reference/idempotency-key).

<Step>
## Create the charge
To create the charge, you must send a `POST` request to the [`/v1/charges`](/docs/api-reference/charges/create-charge) endpoint. You must specify the transaction amount, currency, and payment method.
For the payment method, you can use:
* `source_data`: Raw card details, APMs, or encrypted payloads from Apple Pay and Google Pay.
* `source`: A previously saved payment method identifier.

The following example shows how to create a charge using raw card data. Amounts are specified in the smallest currency unit, so `8000` equals $80.00 USD.

## Handle the response

When you create a charge, the response includes a `status` field that tells you whether the payment requires further action. The following values are possible:

* `succeeded`: The charge was approved instantly. The funds are captured and transferred to the merchant account. You can consider the transaction completed.
* `failed`: The transaction was declined.
* `pending`: Used for asynchronous methods like SEPA or ACH. The charge waits for bank confirmation.
* `requires_authentication`: The transaction requires Strong Customer Authentication (SCA) to proceed. This occurs if a [3DS challenge](/payments/server-to-server-guides/create-a-charge-with-3ds) is required by regional regulations, if you actively enforce 3DS for fraud prevention, or if the customer is using an Alternative Payment Method (APM) that requires a direct bank login.
</Step>

## Handle customer authentication

If the charge response returns `requires_authentication`, redirect the customer to a bank challenge before the funds can be captured.

Send a `POST` request to [`/v1/sessions/authentication/linked`](/docs/api-reference/authenticationsessions/create-linked-authentication-session) with the charge ID in the `resource` field. The `confirm` value controls what happens after the customer completes the challenge.

## Listen for webhooks
Because status updates can happen asynchronously, use [webhooks](/docs/api-reference/webhooks) to trigger your fulfillment logic.
Subscribe to the following events:
* `charge.succeeded`: The funds are successfully captured.
* `charge.failed`: The transaction could not be completed.

<Callout type="warn">
Always verify webhook signatures using the `Pay-Signature` header and your endpoint's secret key (via HMAC SHA-256) to ensure the event is legitimate.
</Callout>
</Step>
