Pay.com lets you charge customers on a recurring basis using stored payment method tokens.
This flow is designed for subscription businesses where the customer is present for the first payment, and all future charges happen automatically without the customer needing to take any action.

## Requirements

Before processing recurring charges, ensure the following:

* You have completed an initial charge with `setup_future_usage: "off_session"` to store the card. See [Create a charge and save card](#step-1-set-up-the-payment-method).
* You have the `payment_method` ID (`pm_card_...`) returned by that initial charge.
* You have the `underlying_network_id` returned by that initial charge, if you plan to use methods 3 or 4 below.

<Steps>
<Step>
## Set up the payment method

Before you can create recurring charges, you must store the customer's card during the first payment. To do this, include `"setup_future_usage": "off_session"` inside your `source_data` object when creating the initial charge.

This tells the issuing bank that the card will be used for future merchant-initiated transactions. The response from that charge contains two values you will need for all future recurring charges:

* `payment_method` — the stored card ID (`pm_card_...`)
* `underlying_network_id` — the network transaction ID that links all charges in this recurring sequence

<Callout type="info">
See [Create a charge](/payments/server-to-server-guides/create-a-charge) for the full initial charge flow. Add `"setup_future_usage": "off_session"` to `source_data` to enable card storage.
</Callout>

</Step>
<Step>
## Create a recurring charge

Once the card is stored, send a `POST` request to [`/v1/charges`](/docs/api-reference/charges/create-charge) at any time to charge the customer. Always set `"off_session": true` to tell the issuing bank the customer is not present. This is required for correct transaction flagging and reduces the chance of declines.

Amounts are in the smallest currency unit, so `5000` equals $50.00 USD.

There are four ways to target the stored card. Choose the one that fits how your system stores customer data.

<Step>
## Handle the response

Because these charges happen in the background, your system must handle each response status programmatically.

* `succeeded`: The recurring charge was approved and settled.
* `failed`: The charge was declined, often due to insufficient funds or an expired card.
* `pending`: Used for payment methods that require bank confirmation, such as SEPA or ACH.

</Step>
<Step>
## Listen for webhooks

Because the customer is not present to fix payment issues, use webhooks to trigger automated recovery logic.

* `charge.succeeded`: The payment was confirmed. Use this to keep the subscription active.
* `charge.failed`: The charge was declined. Use this to send recovery emails or retry the payment.

<Callout type="warn">
Always verify webhook signatures using the `Pay-Signature` header and your endpoint's secret key (via HMAC SHA-256) to ensure the event is legitimate.
</Callout>

</Step>
