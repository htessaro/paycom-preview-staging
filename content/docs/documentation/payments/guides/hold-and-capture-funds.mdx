The hold and capture flow, also known as authorization and settlement, lets merchants reserve funds on a customer's card and settle them at a later time.
This is an advanced flow typically used by businesses that only take money when a service is fulfilled, such as hotels, airlines, or e-commerce platforms that ship physical goods.

## Requirements

Before performing a hold and capture, ensure you have the following:

* [Select your environment Base URL](/docs/api-reference/environments).
* [Have access to your API credentials](/docs/api-reference/authentication).
* A unique [`idempotency-key`](/docs/api-reference/idempotency-key) header for each request.

## Create a hold

To reserve funds without capturing them, send a `POST` request to the [`/v1/holds`](/docs/api-reference/holds/create-hold) endpoint. A successful response returns a Hold object with `status: requires_capture`, indicating the funds are reserved on the customer's card but not yet settled. Amounts are in the smallest currency unit, so `10000` equals $100.00 USD.

You can create a hold using raw card data or a stored payment method.

## Capture funds

When you are ready to settle the transaction, send a `POST` request to [`/v1/charges/{hold_id}/capture`](/docs/api-reference/holds/create-capture). Replace `{hold_id}` with the `id` returned when you created the hold.

You can settle the full amount in one call, or capture in multiple partial installments.

<Tabs items={['Full capture', 'Partial capture']}>
  <Tab value="Full capture">

To settle the entire reserved amount, omit the `amount` field. The hold transitions to `status: succeeded` and `amount_capturable` drops to `0`.

    <Tabs items={['Request', 'Response']}>
      <Tab value="Request">

`POST /v1/charges/{hold_id}/capture`

```json
{}
```
      </Tab>
      <Tab value="Response">
```json
{
  "id": "hld_686178331611104256",
  "resource": "hold",
  "amount": 10000,
  "amount_human_readable": "100.00",
  "amount_capturable": 0,
  "created": "2026-03-09T08:00:00.000Z",
  "currency": "usd",
  "description": "Hold for Order #12350",
  "metadata": {},
  "reference": "order-12350",
  "status": "succeeded",
  "test_mode": true
}
```
      </Tab>
    </Tabs>
  </Tab>
  <Tab value="Partial capture">

To settle only part of the reserved amount, set `amount` to the value you want to settle in this capture, not the total or remaining balance. After a partial capture, the hold stays open with `status: requires_capture` and `amount_capturable` is reduced by the captured amount.

To capture in multiple installments, repeat the following steps until `amount_capturable` reaches `0`:

1. Call `GET /v1/holds/{hold_id}` to check the current `amount_capturable`.
2. Send a `POST` to `/v1/charges/{hold_id}/capture` with the `amount` to settle in this installment.

When the final partial capture brings `amount_capturable` to `0`, the hold transitions to `status: succeeded`.

    <Tabs items={['Request', 'Response']}>
      <Tab value="Request">

`POST /v1/charges/{hold_id}/capture`

```json
{
  "amount": 5000
}
```
      </Tab>
      <Tab value="Response">
```json
{
  "id": "hld_686178331611104256",
  "resource": "hold",
  "amount": 10000,
  "amount_human_readable": "100.00",
  "amount_capturable": 5000,
  "created": "2026-03-09T08:00:00.000Z",
  "currency": "usd",
  "description": "Hold for Order #12350",
  "metadata": {},
  "reference": "order-12350",
  "status": "requires_capture",
  "test_mode": true
}
```
      </Tab>
    </Tabs>
  </Tab>
</Tabs>

For tracking, the system triggers a `hold.captured` event for every capture (including multi-capture scenarios). The `hold.succeeded` event fires only on the first successful capture of that hold.

## Cancel a hold

If you decide not to proceed with a transaction, send a `POST` request to [`/v1/holds/{hold_id}/cancel`](/docs/api-reference/holds/hold-cancel) to release the reserved funds back to the customer.

<Callout type="info">
Cancellation always releases the full remaining `amount_capturable`. If you want to release only part of the reserved funds, perform a partial capture first to settle what you need, then call this endpoint to release whatever remains.
</Callout>

## Monitor status and webhooks

Use webhooks to track the lifecycle of the hold and capture process.

* `hold.requires_capture`: Sent when a hold is approved and funds are successfully reserved and awaiting capture.
* `hold.captured`: Sent for every capture you perform. If you partially capture funds from the same hold three times, your system receives this webhook three times.
* `hold.succeeded`: Sent only for the first capture. It tells your system that the initial settlement was successful and does not fire again for subsequent captures against the same hold.
* `hold.canceled`: Sent when the hold is canceled by the merchant or expires.
* `hold.failed`: Sent if the initial attempt to create the hold is declined.

<Callout type="warn">
Always verify webhook signatures using the `Pay-Signature` header and your endpoint's secret key (via HMAC SHA-256) to ensure the event is legitimate.
</Callout>
