This guide covers how to process Apple Pay and Google Pay charges via the Pay.com API without using hosted components. You render the wallet button on your frontend, receive the payment token from the wallet provider, and pass it to Pay.com to complete the charge.

## Integration paths

There are two ways to send wallet data to the API:

* **Pay.com decrypts the payload:** Your frontend passes the raw encrypted blob from the wallet provider directly to Pay.com. Pay.com handles decryption and processing. This requires a real device session and cannot be tested server-side.
* **You decrypt the payload:** You decrypt the wallet token on your own servers and pass the decrypted DPAN (Device Primary Account Number) to Pay.com. This is the path shown in the code examples below.

## Requirements

Before processing wallet payments via API, complete these steps in your dashboard:

* Enable Apple Pay and Google Pay in your dashboard settings.
* For Apple Pay, register and verify your domain. See [Apple's documentation on configuring your environment](https://developer.apple.com/documentation/apple_pay_on_the_web/configuring_your_environment) for instructions.
* For Google Pay, set the gateway to `paycom` and the `gatewayMerchantID` to `mrchnt_{your_merchant_id}`. See the [Google Pay Web Developer Documentation](https://developers.google.com/pay/api/web/overview) for configuration details.

## Send the charge request

Use the native Apple Pay or Google Pay SDK on your frontend to prompt the customer for payment. Once the customer authorizes, the wallet provider returns a payment object to your client-side code. Pass the token data from that object to your backend, then call the [`POST /v1/charges`](/docs/api-reference/charges/create-charge) endpoint.

Set `source_data.type` to `"network_token"` and provide the decrypted token details inside `source_data.network_token`. Amounts are in the smallest currency unit, so `10000` equals $100.00 USD.

Always pass billing details from the wallet's response into `source_data.billing_details` to improve approval rates.

<Callout type="info">
To use the Pay.com-side decryption path instead, set `source_data.type` to `"applepay"` or `"googlepay"` and pass the full encrypted object from the wallet provider. For Google Pay this is `{ signature, intermediateSigningKey, protocolVersion, signedMessage }`. This path requires a real device session and cannot be tested server-side.
</Callout>

## Handle authentication types

Wallet transactions use one of two authentication methods, depending on how you configure your wallet request.

* **Cryptogram 3DS:** The transaction is fully authenticated by the wallet provider using device-level biometrics (Face ID, Touch ID, or fingerprint). The `eci` value is `05` for Apple Pay and `07` for Visa-network Google Pay. No additional 3DS challenge is needed. This is the recommended method.
* **PAN_ONLY (Google Pay only):** The wallet shares the underlying card number without device-level authentication. Because the card is not biometrically verified, the issuer may require a standard 3DS challenge. If the charge returns `status: requires_authentication`, follow the [3DS challenge flow](/documentation/payments/guides/create-a-charge-with-3ds) to complete the transaction. You must also include `authentication_context` with browser details in your charge request when using PAN_ONLY.

</Step>
<Step>
## Listen for webhooks

Subscribe to these events to track the outcome of each wallet charge:

* `charge.succeeded`: The funds were captured successfully.
* `charge.failed`: The charge was declined by the issuer or wallet provider.

<Callout type="warn">
Always verify webhook signatures using the `Pay-Signature` header and your endpoint's secret key (via HMAC SHA-256) to ensure the event is legitimate.
</Callout>
