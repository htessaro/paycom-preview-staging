Use the Pay.com API to charge a customer's stored payment method without
them being present. This approach gives you full control over when each
charge fires, the amount, and retry logic.

If you want Pay.com to manage the billing schedule automatically, see
[create a subscription](/documentation/payments/guides/create-a-subscription)
instead.

## Requirements

- A stored payment method ID (`pm_card_xxx`) from a previous on-session
  charge with `"setup_future_usage": "off_session"`. See
  [save a payment method](/documentation/payments/guides/save-a-payment-method).
- [Retrieve your API credentials](/docs/api-reference/authentication)
  from your dashboard.
- [Generate a unique `idempotency-key` header](/docs/api-reference/idempotency-key)
  for each request.
- The `underlying_network_id` from the initial charge if you have
  multiple active recurring sequences on the same card.

When you created the initial charge with `"setup_future_usage": "off_session"`,
the response included two values you need for future recurring charges:

- `payment_method` — the stored card ID (`pm_card_...`)
- `underlying_network_id` — the network transaction ID that links all
  charges in this recurring sequence

<Callout type="info">
  See [create a charge](/documentation/payments/guides/create-a-charge)
  for the initial charge flow. Add `"setup_future_usage": "off_session"`
  to `source_data` to enable card storage.
</Callout>

## Create the recurring charge

Send a `POST` request to [`/v1/charges`](/docs/api-reference/charges/create-charge)
with `"off_session": true`. Amounts are in the smallest currency unit
— `5000` equals $50.00 USD.

There are four ways to target the stored card. Choose the one that
fits how your system stores customer data.

<Tabs items={['By payment method ID', 'By customer reference ID', 'By payment method + network ID', 'By raw card data (PCI)']}>
  <Tab value="By payment method ID">
    Pass the `pm_card_...` ID in `source`. Pay.com looks up the
    `underlying_network_id` automatically. Use this when you store
    the payment method ID in your own database.

    <Tabs items={['Request', 'Response']}>
      <Tab value="Request">
        ```json
        {
          "amount": 5000,
          "currency": "usd",
          "source": "pm_card_687080745486778368",
          "off_session": true,
          "description": "Monthly subscription renewal",
          "reference": "sub-renewal-pm-001"
        }
        ```
      </Tab>
      <Tab value="Response">
        ```json
        {
          "id": "chrg_687080841926412288",
          "resource": "charge",
          "amount": 5000,
          "amount_human_readable": "50.00",
          "currency": "usd",
          "status": "succeeded",
          "payment_method": "pm_card_687080745486778368",
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "off_session": true,
          "test_mode": true
        }
        ```
      </Tab>
    </Tabs>
  </Tab>

  <Tab value="By customer reference ID">
    Pass `customer_reference_id` and Pay.com charges the customer's
    default or most recently used stored card. This is the simplest
    approach when a customer has a single stored card.

    <Tabs items={['Request', 'Response']}>
      <Tab value="Request">
        ```json
        {
          "amount": 5000,
          "currency": "usd",
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "off_session": true,
          "description": "Monthly subscription renewal",
          "reference": "sub-renewal-cust-001"
        }
        ```
      </Tab>
      <Tab value="Response">
        ```json
        {
          "id": "chrg_687080913841949696",
          "resource": "charge",
          "amount": 5000,
          "amount_human_readable": "50.00",
          "currency": "usd",
          "status": "succeeded",
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "off_session": true,
          "test_mode": true
        }
        ```
      </Tab>
    </Tabs>
  </Tab>

  <Tab value="By payment method + network ID">
    Pass the payment method ID inside `source_data.source.id` together
    with the `underlying_network_id` from the initial charge. Use this
    when a customer has multiple active recurring sequences on the same
    card and you need to specify which sequence to continue.

    <Tabs items={['Request', 'Response']}>
      <Tab value="Request">
        ```json
        {
          "amount": 5000,
          "currency": "usd",
          "off_session": true,
          "source_data": {
            "type": "source",
            "source": {
              "id": "pm_card_687080745486778368"
            },
            "underlying_network_id": "dcc8d869-fa23-4e4c-ae3e-8e1ec33b0004"
          },
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "description": "Monthly subscription renewal",
          "reference": "sub-renewal-pm-nid-001"
        }
        ```
      </Tab>
      <Tab value="Response">
        ```json
        {
          "id": "chrg_687081737364179968",
          "resource": "charge",
          "amount": 5000,
          "amount_human_readable": "50.00",
          "currency": "usd",
          "status": "succeeded",
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "test_mode": true
        }
        ```
      </Tab>
    </Tabs>
  </Tab>

  <Tab value="By raw card data (PCI)">
    Pass the full card details inside `source_data.card` together
    with the `underlying_network_id` from the initial charge. Use
    this only if you store raw card data on your own servers.

    <Callout type="warn">
      Raw card data passes through your servers. This method requires
      PCI Level 1 or SAQ-D compliance.
    </Callout>

    <Tabs items={['Request', 'Response']}>
      <Tab value="Request">
        ```json
        {
          "amount": 5000,
          "currency": "usd",
          "off_session": true,
          "source_data": {
            "type": "card",
            "card": {
              "number": "4242424242424242",
              "expiry_month": "12",
              "expiry_year": "2028",
              "cvv": "123",
              "name": "Jane Doe"
            },
            "underlying_network_id": "dcc8d869-fa23-4e4c-ae3e-8e1ec33b0004"
          },
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "description": "Monthly subscription renewal",
          "reference": "sub-renewal-chd-nid-001"
        }
        ```
      </Tab>
      <Tab value="Response">
        ```json
        {
          "id": "chrg_687081785254740992",
          "resource": "charge",
          "amount": 5000,
          "amount_human_readable": "50.00",
          "currency": "usd",
          "status": "succeeded",
          "customer_reference_id": "cust-jane-doe-recurring-001",
          "test_mode": true
        }
        ```
      </Tab>
    </Tabs>
  </Tab>
</Tabs>

<Step>
## Handle the response

Because recurring charges happen in the background, your system must
handle each response status programmatically.

- `succeeded`: The charge was approved and settled.
- `failed`: The charge was declined, often due to insufficient funds or
  an expired card. Trigger a retry or prompt the customer to update their
  payment method.
- `pending`: Used for payment methods that require bank confirmation,
  such as SEPA or ACH.
</Step>

</Steps>

## Listen for webhooks

Because the customer is not present during recurring billing, subscribe
to the following events to automate fulfillment and recovery logic on
your backend:

- `charge.succeeded`: The payment was confirmed. Use this to keep the
  subscription active.
- `charge.failed`: The charge was declined. Use this to send recovery
  emails or retry the payment.

<Callout type="warn">
  Always verify webhook signatures using the `Pay-Signature` header and
  your endpoint's secret key via HMAC SHA-256 before acting on any event.
</Callout>
